# 01 - 需求分析

## 1. 项目背景与目标

### 1.1 项目背景

本系统为丽水学院短学期实践作业项目。传统校园报修流程依赖纸质登记或电话沟通，存在以下痛点：

- 报修渠道单一，师生提交不方便
- 工单流转不透明，无法追踪处理进度
- 派单依赖人工协调，效率低下
- 维修结果缺乏反馈机制
- 数据统计困难，无法分析报修趋势

### 1.2 项目目标

构建一套完整的校园工单管理系统，实现报修流程的数字化、透明化、可追溯，覆盖"师生提交报修 → 管理员受理 → 管理员派单 → 维修人员处理 → 报修人确认/反馈"的全链路闭环。

## 2. 核心业务流程

系统核心业务由5个关键动作串联完成：

### 动作一：提交报修（学生/教师）

```
[报修人] → 选择报修分类 → 填写标题/描述/位置 → 上传图片(可选) → 提交工单
   ↓
[系统] → 创建工单(状态:PENDING) → 记录日志 → 通知管理员
```

### 动作二：受理工单（管理员）

```
[管理员] → 查看待处理工单列表 → 点击受理
   ↓
[系统] → 状态变更 PENDING → ACCEPTED → 记录日志 → 通知报修人
```

### 动作三：派发工单（管理员）

```
[管理员] → 查看已受理工单 → 选择维修人员 → 确认派单
   ↓
[系统] → 状态变更 ACCEPTED → ASSIGNED → 记录日志 → 通知维修人员
```

### 动作四：处理工单（维修人员）

```
[维修人员] → 查看已分配工单 → 开始处理 → 到场维修 → 提交处理结果(上传维修后照片)
   ↓
[系统] → 状态变更 ASSIGNED → PROCESSING → PENDING_CONFIRM → 记录日志 → 通知报修人
```

### 动作五：确认/反馈（报修人）

```
[报修人] → 查看待确认工单 → 查看维修结果
   ├─ 确认完成 → 状态变更 PENDING_CONFIRM → COMPLETED
   └─ 反馈未解决 → 状态变更 PENDING_CONFIRM → UNRESOLVED
   ↓
[系统] → 记录日志 → 通知管理员/维修人员
```

## 4. 工单状态流转说明

### 4.1 状态定义（8个状态）

| 状态 | 常量值 | 含义 | 停留角色 |
|------|--------|------|----------|
| 待受理 | `PENDING` | 师生已提交，等待管理员受理 | 管理员 |
| 已受理 | `ACCEPTED` | 管理员已受理，等待派单 | 管理员 |
| 已派单 | `ASSIGNED` | 已分配给维修人员，等待处理 | 维修人员 |
| 处理中 | `PROCESSING` | 维修人员已开始处理 | 维修人员 |
| 待确认 | `PENDING_CONFIRM` | 维修人员已提交结果，等待报修人确认 | 报修人 |
| 已完成 | `COMPLETED` | 报修人确认完成 | 终态 |
| 未解决 | `UNRESOLVED` | 报修人反馈未解决，需重新派单 | 管理员 |
| 已取消 | `CANCELLED` | 管理员取消工单 | 终态 |

### 4.2 状态转换规则

```
                              ┌────────────────┐
                              │    PENDING      │  ← 学生/教师提交报修
                              │    (待受理)      │
                              └───────┬────────┘
                                      │ 管理员受理
                              ┌───────▼────────┐
                              │    ACCEPTED     │
                              │    (已受理)      │
                              └───────┬────────┘
                                      │ 管理员派单
                              ┌───────▼────────┐
                              │    ASSIGNED     │
                              │    (已派单)      │
                              └───────┬────────┘
                                      │ 维修人员开始处理
                              ┌───────▼────────┐
                              │   PROCESSING    │
                              │    (处理中)      │
                              └───────┬────────┘
                                      │ 维修人员提交结果
                              ┌───────▼────────┐
                              │PENDING_CONFIRM  │  ←──────────────────┐
                              │   (待确认)       │                     │
                              └───┬──────┬──────┘                     │
                                  │      │                             │
                    报修人确认完成  │      │ 报修人反馈未解决              │
                                  │      │                             │
                    ┌─────────────▼┐   ┌─▼────────────┐                │
                    │  COMPLETED   │   │  UNRESOLVED   │─── 返回 ASSIGNED
                    │  (已完成)     │   │  (未解决)      │    (重新派单)
                    └──────────────┘   └───────────────┘

    ┌──────────┐   管理员取消
    │ PENDING  │──────┐
    │ ACCEPTED │──────┤
    │ ASSIGNED │──────┤
    └──────────┘      │
               ┌──────▼──────┐
               │  CANCELLED  │
               │  (已取消)    │
               └─────────────┘
```

### 4.3 转换约束

- `PENDING` → 仅管理员可受理 → `ACCEPTED`
- `ACCEPTED` → 仅管理员可派单 → `ASSIGNED`
- `ASSIGNED` → 仅被指派的维修人员可开始处理 → `PROCESSING`
- `PROCESSING` → 仅被指派的维修人员可提交结果 → `PENDING_CONFIRM`
- `PENDING_CONFIRM` → 仅报修人本人可确认完成 → `COMPLETED`
- `PENDING_CONFIRM` → 仅报修人本人可反馈未解决 → `UNRESOLVED`
- `UNRESOLVED` → 管理员可重新派单 → `ASSIGNED`
- `PENDING`/`ACCEPTED`/`ASSIGNED` → 管理员可取消 → `CANCELLED`
- `COMPLETED`/`CANCELLED` → 终态，不可再变更

## 6. 非功能需求

### 6.1 性能需求

| 指标 | 要求 |
|------|------|
| 页面加载时间 | 首屏 < 2s |
| API响应时间 | P95 < 500ms |
| 并发用户数 | 支持200人同时在线 |
| 文件上传 | 支持最大10MB图片上传 |

### 6.2 安全需求

| 项目 | 说明 |
|------|------|
| 认证方式 | JWT Bearer Token |
| Token有效期 | 7天 |
| 密码加密 | bcrypt 加盐哈希 |
| 权限控制 | 后端RBAC + 数据权限校验 |
| 接口防护 | NestJS Guard + 角色装饰器 |
| 敏感数据 | 密码脱敏，不在响应中返回 |

### 6.3 可用性需求

- 系统界面适配PC端，支持主流浏览器（Chrome、Edge、Firefox）
- 提供清晰的错误提示和操作确认
- 操作日志完整记录，便于追溯
- 通知实时推送，减少页面刷新

---

## 7. 加分项功能

### 7.1 移动端适配

- 前端采用响应式布局，适配手机/平板访问
- 核心操作（查看工单、提交报修、确认完成）在移动端流畅可用

### 7.2 图片压缩

- 前端上传前对图片进行压缩处理，减小文件体积
- 减少服务器存储压力，加快上传速度

### 7.3 数据大屏

- 管理员专属可视化面板
- 展示：今日新增工单数、各状态工单分布（饼图）、近7天工单趋势（折线图）、各分类工单统计（柱状图）、维修人员处理效率排行
